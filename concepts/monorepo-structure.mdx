The Gigadrive SDK is organized as a **pnpm workspace monorepo** containing the CLI, shared libraries, UI components, and build utilities. This structure enables code sharing, unified tooling, and consistent versioning across all packages.

## Repository Overview

The monorepo contains **7 packages** organized under `packages/` and `local/`:

```bash
packages/sdk              # @gigadrive/sdk - Core SDK (stub)
packages/commons          # @gigadrive/commons - Shared utilities
packages/network-config   # @gigadrive/network-config - Config parsing
packages/build-utils      # @gigadrive/build-utils - Build utilities
packages/cli              # gigadrive - CLI tool
packages/harmony          # @gigadrive/harmony - React UI components
local/tsup                # @local/tsup - Shared build configs (private)
```

<Info>
The `local/` directory contains workspace-private packages that are not published to npm.
</Info>

## Workspace Configuration

### pnpm Workspace

The repository uses **pnpm workspaces** (v10.28+) for dependency management:

```yaml pnpm-workspace.yaml
packages:
  - 'packages/*'
  - 'local/*'
```

This enables:
- **Workspace dependencies:** Packages reference each other with `workspace:*`
- **Hoisting:** Common dependencies are hoisted to the root `node_modules`
- **Atomic installs:** All packages install together

### Turborepo

Build orchestration uses **Turborepo** for:
- **Parallel builds:** Packages build in parallel based on dependency graph
- **Incremental builds:** Only rebuild changed packages
- **Remote caching:** CI caches build outputs

```bash
pnpm build              # Build all packages
pnpm build:no-cache     # Skip Turbo cache
pnpm clean              # Remove all dist/ directories
```

## Package Structure

All packages follow a consistent structure:

```bash
packages/example/
  src/
    index.ts           # Barrel export — re-exports all public modules
    feature-a.ts       # Feature implementation
    feature-a.test.ts  # Co-located tests (Vitest)
    feature-b.ts
    feature-b.test.ts
  dist/                # Build output (gitignored)
    index.js           # CommonJS
    index.mjs          # ESM
    index.d.ts         # TypeScript declarations
  package.json
  tsconfig.json
  tsup.config.ts       # Build config (imports @local/tsup)
```

<Tip>
All test files (`*.test.ts`) are co-located with their source files and excluded from the build output.
</Tip>

## Cross-Package Imports

Packages reference each other using scoped imports:

```typescript
// In packages/cli/src/services/auth.ts
import { formatBytes } from '@gigadrive/commons';
import { parseConfig } from '@gigadrive/network-config';
import { exec } from '@gigadrive/build-utils';
```

**Within a package**, use relative imports:

```typescript
// In packages/commons/src/index.ts
export * from './format-bytes';
export * from './deep-merge';
```

## Build System

### Dual Module Output

All packages output **dual ESM + CJS** for maximum compatibility:

```json package.json
{
  "main": "./dist/index.js",       // CommonJS entry
  "module": "./dist/index.mjs",    // ESM entry
  "types": "./dist/index.d.ts",    // TypeScript declarations
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.js",
      "types": "./dist/index.d.ts"
    }
  }
}
```

### Shared Build Config

Most packages use the shared tsup config:

```typescript tsup.config.ts
import { index } from '@local/tsup';
export default index;
```

This provides:
- ✅ CJS, ESM, and IIFE output
- ✅ TypeScript declarations
- ✅ Source maps
- ✅ All dependencies marked external
- ✅ Tree-shaking enabled

<Note>
The `harmony` UI library uses **Rslib** instead of tsup for unbundled component builds.
</Note>

## Version Management

The monorepo uses **Changesets** for version management:

```bash
pnpm changeset          # Create a changeset (version bump intent)
pnpm changeset version  # Apply changesets to package.json files
pnpm changeset publish  # Publish changed packages to npm
```

Publishing is **automated via GitHub Actions** on push to `main`.
